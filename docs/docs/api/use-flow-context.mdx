# useFlowContext

Hook for **chrome components** and any consumer component inside the outlet's provider boundary. Returns the current flow context value plus `resolve` and `abort` callbacks for flow termination.

Has **no navigation capabilities** — it cannot `advance` or `retreat`. Navigation remains exclusive to step components via `useStep`.

## Signature

```typescript
function useFlowContext<TContext = unknown>(): {
  context: TContext;
  resolve: (value?: unknown) => void;
  abort: (reason?: unknown) => void;
};
```

## Returns

| Property | Type | Description |
|---|---|---|
| `context` | `TContext` | The current consumer-owned flow context value. |
| `resolve` | `(value?: unknown) => void` | End the flow successfully. The value is passed to the promise returned by `initFlow`. |
| `abort` | `(reason?: unknown) => void` | End the flow without completing. The reason is passed as a rejection to the `initFlow` promise. |

## When to Use

`useFlowContext` exists for **non-step** components that need to respond to flow state or terminate a flow — most commonly chrome components like modal headers, progress bars, or close buttons.

Steps should use `useStep()` instead. `useStep` provides the same `resolve`/`abort` plus navigation (`advance`/`retreat`) and context access that steps need.

## Chrome Pattern

The primary use case: a chrome component reads context to display dynamic content that stays stable across step transitions, and provides a close button that aborts the flow.

```tsx
// Chrome component — stays mounted across step transitions
function ModalHeader() {
  const { context: ctx, abort } = useFlowContext<{ title: string; step: number }>();
  return (
    <header>
      <h2>{ctx.title}</h2>
      <span>Step {ctx.step} of 3</span>
      <button onClick={() => abort("cancelled")}>✕</button>
    </header>
  );
}

// Step — patches context to update the chrome
function Step1() {
  const { advance } = useStep();
  return (
    <div>
      <p>Welcome to the flow!</p>
      <button
        onClick={() =>
          advance(Step2, { title: "Personal Info", step: 2 })
        }
      >
        Next
      </button>
    </div>
  );
}

// Outlet — chrome child is outside Suspense, inside provider
function Host() {
  const ref = useRef<FlowOutletHandle>(null);
  const { initFlow } = useFlowInit();

  const start = () => {
    initFlow(Step1, ref, { title: "Welcome", step: 1 });
  };

  return (
    <div>
      <FlowOutlet
        ref={ref}
        chrome={(children) => (
          <>
            <ModalHeader />
            {children}
          </>
        )}
      />
      <button onClick={start}>Start</button>
    </div>
  );
}
```

### Why chrome stays stable

The chrome component (`<ModalHeader>` above) is rendered **inside** the outlet's context provider but **outside** the Suspense boundary. This means:

- Chrome re-renders when context changes (via `contextPatch`) — so the title updates immediately.
- Chrome does **not** unmount or flicker when an async step is loading — the Suspense fallback only affects the step area.

```
<FlowContext.Provider>           ← flow-level context (useFlowContext reads this)
  {chrome(
    <StepContext.Provider>       ← step-only context (useStep reads this)
      <FlowErrorBoundary>
        <Suspense fallback={…}>   ← only wraps the step
          <ActiveStep />
        </Suspense>
      </FlowErrorBoundary>
    </StepContext.Provider>
  )}
  // inside chrome:
  <ModalHeader />              ← reads useFlowContext, stable across transitions
</FlowContext.Provider>
```

### Why chrome gets `resolve`/`abort` but not `advance`/`retreat`

Chrome is stable UI that wraps the step area. It should be able to **terminate** a flow (e.g. a close button that aborts, or a "finish" action that resolves) but must **not drive navigation** between steps — that responsibility belongs to the step itself, per the "step decides what comes next" paradigm.

This separation is structurally enforced, not just a convention. `advance` and `retreat` live in the inner `StepContext`, which is only present inside the active step's subtree. Chrome components sit inside the outer `FlowContext` (which provides this hook's values) but outside `StepContext`. Calling `useStep()` from chrome throws immediately at render time.

If chrome needs to influence which step renders next, the step should read a signal from context and branch accordingly.

## Generic: `TContext`

```tsx
interface MyFlowContext {
  title: string;
  step: number;
  canClose: boolean;
}

function ModalChrome() {
  const { context: ctx, abort } = useFlowContext<MyFlowContext>();
  // ctx is typed as MyFlowContext
  return (
    <header>
      {ctx.title}
      {ctx.canClose && <button onClick={() => abort()}>✕</button>}
    </header>
  );
}
```

## Error handling

Throws if called outside a `<FlowOutlet />` provider boundary:

```
useFlowContext must be used within a <FlowOutlet /> provider boundary
```

This means `useFlowContext` can only be called from components rendered as children of `<FlowOutlet>` (chrome) or from step components rendered inside the outlet.
